# 비디오 얼굴 감지 개선 작업 노트

작성일: 2026-07-06 · 브랜치: `improve-video-detection` · 테스트 영상: MOT17-11 (두바이몰, 1080p/30fps/900프레임)

## 1. 배경 — 보고된 증상

2.5G(Fast) 모델로 비디오를 처리할 때:

1. 마스크가 얼굴이 도착하기 몇 프레임 전에 미리 그 자리에 가 있음
2. 얼굴이 아닌 곳(LG 로고 등 간판)에 마스크가 붙음
3. 충분히 큰 얼굴인데 감지를 못함

## 2. 진단 결과

### 근본 원인: 감지가 원본의 1/3 해상도에서 실행됨

`2.5g_bnkps.onnx`는 입력이 **640×640 고정**으로 export된 모델이다(10G는 동적).
앱은 `kVideoDetectionInputSize = 960`으로 요청하지만 `ScrfdFaceDetector` 생성자가
모델의 고정 크기로 조용히 덮어써서, 1080p 기준 실제 감지 입력은
1920→960(ffmpeg, `analysisLongEdge`)→640(레터박스) = **원본의 1/3**이었다.
로그는 "960 px"로 표시되어 사실과 달랐다.

MOT17-11 실측: 900프레임에서 점수 0.1 이상 감지 15,709개 중 **0.5 이상은 88개(0.6%)**.
거의 모든 얼굴 점수가 0.2~0.49에 눌려 있었고, 이것이 세 증상을 모두 만들었다.

### 증상별 메커니즘

| 증상 | 메커니즘 |
|---|---|
| 마스크 선행 | 감지가 자주 끊김 → 최대 30프레임 직선 보간 + 트랙 양끝 3프레임 정적 연장이 "추측"으로 메꿈. 감지가 전혀 없는 곳의 마스크(고스트)가 15개 트랙, 최장 연속 36프레임 |
| 간판 오탐 | 트랙 시작은 0.5 한 프레임이면 충분, 유지는 0.1이면 충분 + 30프레임 무감지 허용. LG 기둥이 한 번 0.50을 넘자 121프레임(영상 끝까지) 마스킹됨. LG 광고 배너 72프레임, 전광판 26프레임 |
| 얼굴 미감지 | 트랙 시작에 사용자 임계값(0.5) 이상이 최소 1프레임 필요한데, 실제 얼굴들이 저해상도 탓에 끝까지 0.5 미만. 예: 326프레임 동안 최고 0.498로 감지되고도 한 번도 마스킹 안 된 얼굴, 원본 74×98px 정면 얼굴이 최고 0.434 |

## 3. 완료된 작업

### f9ebf50 — 고정 입력 SCRFD 모델을 요청 크기로 실행

- `OnnxGraphPatch` 신설(`makeOnnxSpatialDimsDynamic`): ONNX protobuf를 직접 파싱해
  입력 공간 차원을 동적으로 바꾸고, FPN 업샘플 Resize 노드 2개를 고정 sizes → 2배 scales로 교체.
  외부 라이브러리 의존 없음.
- `ScrfdFaceDetector`: 모델을 메모리로 읽어 세션 생성. 고정 크기 ≠ 요청 크기면 패치 시도 →
  프로브 추론으로 검증 → 성공 시 채택, 실패 시 모델 고유 크기로 폴백.
  가속(CoreML 등) 상태에서 프로브가 실패하면 예외를 올려 기존 CPU 폴백 경로(10G와 동일)를 타게 함.
- **배포 변경 없음**: 다운로드 URL·체크섬 그대로, 변환은 사용자 기기 메모리 안에서만 일어남
  (InsightFace 모델은 비상업 연구용이라 수정본 재배포는 라이선스상 부적절 — 검토 후 로컬 변환으로 결정).
- 로그가 요청 크기가 아닌 **실제 사용 크기**를 출력하도록 수정, 한국어 번역 2건 갱신.
- 검증: 패치 전후 640 입력 출력 차이 3×10⁻⁶(부동소수점 노이즈), 960에서 앵커 수 정확히 일치.

### 419d253 — 트랙 품질 필터와 저신뢰 연장 제한

- **품질 필터**: 강한 감지(시작 임계값 이상, 비보간) 3개 미만인 트랙 제거 (`minStrongDetections = 3`)
- **시작 히스테리시스**: 트랙 시작 임계값 = `min(사용자 임계값, max(0.35, 임계값 − 0.1))` — 기본 0.4.
  경계선 얼굴(0.4~0.5)을 구제하되 마스킹 여부 판단은 트랙 단위 근거로.
- **코스팅 제한**: `maxFramesSinceHighScore = 30` — 저점수 매칭만으로는 30프레임까지만 연장 후 트랙 종료
- **보간 갭 축소**: `maxInterpolationGap` 30 → 12

### 검증 결과 (MOT17-11, 파이프라인 파이썬 재현 실측)

| 지표 | 개선 전 (640+기존) | 개선 후 (960+신규) |
|---|---|---|
| 점수 0.5+ 감지 | 88 | 571 |
| 최종 트랙 수 | 24 | 54 |
| 고스트 마스크 | 15개 트랙, 최장 36프레임 연속 | 1개 트랙 30프레임 |
| LG 광고 배너 오탐 | 72프레임 | 0 (최고 점수 0.28로 하락) |
| LG 기둥 로고 오탐 | 121프레임(영상 끝까지) | 약 80프레임으로 감소, 코스팅 제한으로 종결 |
| 326프레임 미감지 얼굴(최고 0.498) | 미마스킹 | 마스킹됨 |
| 원본 74×98px 얼굴 | 미마스킹 | 마스킹됨 |

단위/통합 테스트: 4개 스위트 전부 통과. 신규 테스트 — 저신뢰 코스팅 만료, 약한 트랙 제거,
패처의 비정상 입력 거부, 실제 2.5G 모델 320px 변환·추론(모델 파일이 `models/`에 있을 때만 실행).

## 4. 남은 작업

### 필수

- [ ] **실기기 E2E 확인**: 실제 앱으로 MOT17-11 처리해 체감 확인.
  특히 CoreML이 동적 2.5G를 거부할 때 CPU 폴백 경로("GPU acceleration can't run the video face
  model…" 로그)가 실기기에서 정상 동작하는지. 처리 속도 변화(960 감지는 640 대비 연산 ~2.25배)도 확인.
- [ ] Windows(DirectML)·Linux에서 동적 패치 모델 동작 확인.
- [ ] `main` 머지 및 릴리스 노트(RELEASE_NOTES.md) 갱신.

### 후속 개선 (별도 작업)

- [ ] **LG 로고류 잔존 오탐**: 로고 도상이 실제로 얼굴 형태(원 안의 눈·코)라 960에서도 최고 0.54가 나옴.
  트래킹 파라미터로는 완전 제거 불가. bnkps 모델이 이미 출력하는 5점 랜드마크(kps)의 기하 일관성
  검증(눈 간격/코 위치 비율 등)을 후처리에 추가하는 것이 정공법. 현재 C++ 디코더는 kps 출력을 버리고 있음.
- [ ] **미스링크 드리프트 잔존**: 트랙이 저점수 매칭으로 인접 물체(전광판 등)로 옮겨 타는 사례가
  줄었지만 남음(실측 1건, 38프레임). `gapMotionTooFast` 임계값(프레임당 박스 크기 0.75배) 축소와
  저점수 매칭 IoU 기준 상향 검토.
- [ ] **초소형 얼굴**: 원본 30px 미만 얼굴은 960에서도 간헐적으로 놓침. `analysisLongEdge` 상향(연산
  비용 트레이드오프) 또는 놓친 구간 한정 재감지 패스 검토.
- [ ] **번역 정식 갱신**: 이 머신에 qttools(lupdate)가 없어 `redactly_ko.ts`를 직접 수정함.
  `location` 라인 메타데이터가 낡았으므로 lupdate 실행 환경에서 한 번 재생성 권장.
- [ ] **튜닝 상수 검토**: `minStrongDetections=3`, `maxFramesSinceHighScore=30`, 히스테리시스 0.35 하한,
  보간 갭 12는 MOT17-11 한 영상 기준. 다른 유형(고정 카메라, 근접 인물, 저조도) 영상으로 교차 검증 필요.

## 5. 참고 — 재검증 방법

세션 스크래치패드(임시)에 진단 도구가 있었음: `detect_dump.py`(프레임별 감지 덤프),
`analyze.py`(트래커 파이썬 재현 + 고스트/미감지/오탐 리포트), `viz.py`(프레임 렌더),
`patch_dynamic.py`(모델 패치 검증). 재현이 필요하면 같은 구조로 다시 만들면 된다 —
핵심은 앱과 동일한 전처리(960 긴 변 축소 → 모델 입력 레터박스)와 동일한 트래커 상수를 쓰는 것.
